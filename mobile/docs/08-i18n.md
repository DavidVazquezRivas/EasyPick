# I18n

Guia rapida para internacionalizacion en frontend.

## Estado actual

1. Librerias: i18next, react-i18next, expo-localization, async-storage.
2. Idiomas activos: es y en.
3. Deteccion inicial: idioma del dispositivo.
4. Persistencia: AsyncStorage.

## Estructura

```txt
src/core/i18n/
  index.ts
  resources/
    en/{common,auth,garments}.ts
    es/{common,auth,garments}.ts
```

## Convencion de claves

Usar anidacion por dominio y vista.

Ejemplo de formato:

- domain.view.section.field
- Ejemplo real: garment.detail.form.name.label

Ejemplos actuales:

- common.app.name
- auth.login.subtitle
- auth.login.errors.signInFailed
- garment.home.title

Para errores de backend por code (ApiResponse.message.code), usar:

- common.api.errors.backendCodes.<code>
- Ejemplo: common.api.errors.backendCodes.1001

## Uso en componentes

1. Usar `useTranslation()` sin pasar namespace.
2. Consumir claves directas sin prefijo `archivo:`.

Ejemplo:

```ts
const { t } = useTranslation()
const title = t('garment.home.title')
```

## Reglas

1. Toda cadena visible en UI va a i18n.
2. Agrupar por vista (home, detail, form, etc.).
3. No crear claves planas nuevas.
4. Mantener simetria de claves entre en y es.
5. Si core lanza errores para UI, usar claves i18n (no texto hardcodeado).
6. Cuando backend envie ErrorCode numerico, mapearlo en core/api/global/mapApiError.ts y definir traducciones en en/es.

## Flujo al anadir traducciones

1. Crear clave en en y es.
2. Usar la misma ruta en ambos idiomas.
3. Consumir con `t('domain.view.path')`.
4. Revisar que no queden textos hardcodeados.
